# Repository Instructions

- After every code change, create a git commit and push it to `origin` so deployed environments like `streamlit.app` receive the update.
- Do not leave completed code changes only in the local working tree.
- If a change adds a new import or runtime dependency, update `requirements.txt` in the same commit before pushing.
- After each push, verify the deployed UI shows the new `Build` hash before assuming the change is live.
- For user-reported stats bugs, add or update a regression test with the exact reported numbers before calling the fix complete.
- When changing overview/stat formulas, bump the UI overview schema/version so stale Streamlit session data is forcibly refreshed.
- After any app/UI/filter/behavior change, update the relevant `MD` documentation in the same task so the repository docs stay current.
- UI labels must stay English-only. Use abbreviated labels where space is constrained, but prefer full labels when the layout has room.
- `Hero Overview` and `Detailed Turbo Stats` must stay synchronized on metric coverage, metric order, and value formatting. If a metric is added, removed, reordered, or reformatted in one, apply the same change in the other unless the task explicitly requires divergence.
- For `Hero Overview` specifically, keep column headers abbreviated. For `Detailed Turbo Stats`, use full metric labels where space allows.
- Visual meaning must stay synchronized too: if a metric is color-coded in `Hero Overview` (for example win/loss or winrate semantics), apply the same semantic color treatment in `Detailed Turbo Stats`.
- In every app table or stat block, winrates must always use the same semantic colors: `<50%` red, `=50%` yellow, `>50%` green.
- In every app table or stat block, won-match counts must be green and lost-match counts must be red.
- Exception: in the four `Matchups` tables, apply semantic colors only to `WR`; keep `Won` and `Lost` neutral there.
- Any metric that users may sort by in the UI must stay numeric in the underlying dataframe/state. Never store sortable percentages as strings like `100%`; keep them numeric and apply formatting only in the rendering layer.
- Matchup tables must use one fixed column order everywhere: `Hero Icon`, `Hero`, `WR`, `Matches`, `Won`, `Lost`.
- Any implemented change must be recorded in the relevant project `MD` files in the same task so the next agent can reconstruct prior decisions, behavior changes, cache rules, UI rules, and known constraints from repository docs alone.
- App-wide cache policy: only the main dashboard refresh may perform network sync/hydration. Section refreshes (`Hero Details`, `Matchups`, `Item Winrates`, `Recent Matches`, trends, and derived tables) must rebuild from cached summaries/details only and must not issue hidden OpenDota detail fetches.
- For stats/cache bugs, do not stop at a symptom fix. The required workflow is: centralize the builder path, add/update runtime guard or invariant where possible, add regression coverage for the exact reported data, add at least one cross-path cache test, then update the relevant `MD` docs in the same task.
- Section independence rule: loading or refreshing one selected-hero section must not hide, reset, clear, or collapse any other already loaded selected-hero section for the same active hero snapshot. Section state must restore independently from cache across Streamlit reruns.
- Overview architecture rule: Hero Overview must be built from a validated snapshot that tracks match-summary scope and match-detail coverage together. Do not render or cache overview rows if the snapshot still contains incomplete zero-economy/zero-damage hero rows.
- Item stats architecture rule: Item Winrates must be built from a coverage-aware snapshot. Prefer cached `purchase_log` data, fall back to cached final inventory / summary final slots only when needed, and do not present partial item coverage as fully complete without an explicit warning.
- Snapshot freshness rule: section/session invalidation must follow real dashboard sync timestamps only. Never invalidate section state because local enrichment updated cached rows; enrichment is not a new dashboard snapshot.
